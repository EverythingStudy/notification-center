# 亿级用户消息中心系统设计文档

## 1. 项目目标

设计一个支持亿级用户的消息中心系统，满足以下场景：

### 广播消息

* 系统公告
* VIP公告
* 营销活动通知
* 标签用户通知

特点：

* 一条消息面向大量用户
* 用户数量可达 1 亿+
* 不能采用写扩散方案

---

### 用户消息

* 订单通知
* 物流通知
* 售后通知
* 风控通知

特点：

* 一条消息对应一个用户
* 强关联业务实体
* 需要记录已读状态

---

### 非功能性要求

* 支持 1 亿用户
* 支持水平扩展
* 支持 Kafka 异步投递
* 支持消息撤回
* 支持多端同步
* 支持未读数统计
* 支持高并发查询
* 支持冷热数据分离

---

# 2. 核心设计原则

## 2.1 广播消息与用户消息分离

### 广播消息

采用：

**Fan-Out Read（读扩散）**

即：

消息只存一份。

用户查询时动态计算可见消息。

禁止：

```text
1条广播消息生成1亿条用户记录
```

---

### 用户消息

采用：

**Fan-Out Write（写扩散）**

直接写入用户消息表。

例如：

```text
订单支付成功
```

直接生成：

```text
user_message
```

记录。

---

## 2.2 Feed模型

系统中不存在全局消息流。

采用 Feed 模型。

例如：

```text
SYSTEM_FEED
VIP_FEED
MARKETING_FEED
ORDER_FEED
LOGISTICS_FEED
RISK_FEED
```

不同 Feed 独立维护消息游标。

---

## 2.3 Cursor 单调递增

Cursor 表示：

用户在某个 Feed 中已读到的位置。

Cursor 只能前进。

禁止回退。

更新规则：

```text
newCursor = max(oldCursor,newCursor)
```

---

# 3. 领域模型

## Message

消息主体。

描述消息内容。

---

## Feed

消息流。

定义消息属于哪个可见范围。

例如：

```text
SYSTEM
VIP
MARKETING
ORDER
LOGISTICS
RISK
```

---

## Cursor

用户阅读进度。

记录用户在 Feed 中已读位置。

---

## PushTask

推送任务。

用于 APNs、FCM、厂商 Push。

---

## Subscription

用户订阅关系。

用于判断用户是否可以看到某个 Feed。

例如：

普通用户：

```text
SYSTEM
```

VIP用户：

```text
SYSTEM
VIP
```

---

# 4. 数据库设计

## 4.1 message

消息主体表。

### 字段

| 字段          | 说明               |
| ----------- | ---------------- |
| id          | 消息ID             |
| biz_type    | 业务类型             |
| title       | 标题               |
| content_url | 内容存储地址           |
| send_type   | BROADCAST / USER |
| status      | 状态               |
| create_time | 创建时间             |
| expire_time | 过期时间             |

### 状态

```text
NORMAL
RECALL
EXPIRE
```

### 发送类型

```text
BROADCAST
USER
```

说明：

content_url 指向 OSS 或对象存储。

---

## 4.2 message_feed_mapping

消息与 Feed 映射表。

### 字段

| 字段         | 说明     |
| ---------- | ------ |
| message_id | 消息ID   |
| feed_type  | Feed类型 |

说明：

一条消息可以属于多个 Feed。

例如：

```text
VIP活动
↓
VIP_FEED
```

---

## 4.3 user_message

用户消息表。

仅存储用户级消息。

### 字段

| 字段          | 说明   |
| ----------- | ---- |
| id          | 主键   |
| user_id     | 用户ID |
| message_id  | 消息ID |
| biz_type    | 业务类型 |
| status      | 已读状态 |
| read_time   | 阅读时间 |
| create_time | 创建时间 |

### 唯一索引

```sql
(user_id,message_id)
```

用于保证消费幂等。

---

## 4.4 user_feed_cursor

用户 Feed 游标表。

### 字段

| 字段          | 说明     |
| ----------- | ------ |
| user_id     | 用户ID   |
| feed_type   | Feed类型 |
| cursor      | 已读位置   |
| update_time | 更新时间   |

### 唯一索引

```sql
(user_id,feed_type)
```

---

## 4.5 push_task

推送任务表。

### 字段

| 字段          | 说明   |
| ----------- | ---- |
| id          | 主键   |
| message_id  | 消息ID |
| target_type | 推送目标 |
| status      | 状态   |
| retry_count | 重试次数 |
| create_time | 创建时间 |

---

# 5. Redis设计

## 5.1 用户Cursor

采用 Hash 存储。

### Key

```text
user:{userId}:cursor
```

### Field

```text
system
vip
marketing
order
logistics
risk
```

### 示例

```text
system -> 1200
vip -> 300
marketing -> 88
```

---

## 5.2 Feed最新位置

### Key

```text
feed:{feedType}:max_cursor
```

例如：

```text
feed:system:max_cursor
```

Value：

```text
1250
```

---

## 5.3 未读数计算

```text
feedUnread = feedMaxCursor - userFeedCursor
```

总未读：

```text
totalUnread = Σ(feedUnread)
```

---

# 6. 分库分表设计

## 6.1 user_message

按 user_id 分片。

### 分片规则

```text
user_id % 1024
```

### 表结构

```text
user_message_0000
...
user_message_1023
```

原因：

查询维度始终围绕 user_id。

---

## 6.2 user_feed_cursor

同样按 user_id 分片。

### 分片规则

```text
user_id % 1024
```

---

## 6.3 message

消息总量较小。

无需分片。

按时间归档即可。

---

# 8. 消息查询流程

```text
用户打开消息中心
        ↓
获取用户可访问Feed
        ↓
获取Cursor
        ↓
查询消息
        ↓
过滤RECALL/EXPIRE
        ↓
统一返回
```

返回：

```text
系统消息
VIP消息
订单消息
物流消息
```

统一时间线展示。

# 9. Cursor更新流程

客户端上报：

```text
feedType
cursor
```

服务端更新：

```text
cursor = max(oldCursor,newCursor)
```

同步：

```text
Redis
MySQL
```

原则：

Cursor 永远不能回退。

---

# 10. 多端一致性

采用：

**用户维度 Cursor**

而非设备维度 Cursor。

例如：

```text
userId=1001

systemCursor=1200
vipCursor=300
```

所有设备共享。

设备包括：

```text
iPhone
Android
Web
PC
```

---

## 实时同步

采用：

```text
WebSocket
```

广播：

```json
{
  "type":"cursor_update",
  "feed":"system",
  "cursor":1200
}
```

# 11. 消息撤回设计

禁止：

```sql
delete from message
```

采用：

逻辑删除。

### 状态

```text
RECALL
```

### 操作

```sql
update message
set status='RECALL'
where id=?
```

---

## 在线用户同步

通过 Kafka 发送：

```text
message_recall
```

事件。

客户端收到后：

移除消息展示。

---

# 12. Kafka一致性设计

Kafka保证：

```text
At Least Once
```

不保证 Exactly Once。

---

## 幂等方案

### user_message唯一索引

```sql
unique(user_id,message_id)
```

重复消费直接忽略。

# 13. 冷热数据分离

## 热数据

最近30天。

存储：

```text
MySQL
Redis
```

---

## 冷数据

30天以上。

迁移：

```text
ClickHouse
Elasticsearch
HBase
```

---

## 查询策略

```text
优先查询热数据
历史查询走冷存储
```

---

# 14. 核心设计总结

1. 广播消息采用读扩散（Fan-Out Read）
2. 用户消息采用写扩散（Fan-Out Write）
3. 引入 Feed 模型管理消息流
4. Feed 独立维护 Cursor
5. Cursor 单调递增，不允许回退
6. Redis Hash 存储 Cursor
7. user_message 按 user_id 分库分表
8. Kafka 消费采用业务幂等
9. 消息撤回采用逻辑删除
10. 多端共享 Cursor
11. WebSocket 实现实时同步
12. 热冷数据分层存储
