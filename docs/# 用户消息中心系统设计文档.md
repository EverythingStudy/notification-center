# 用户消息中心系统设计文档

## 1. 项目目标

设计一个支持亿级用户的消息中心系统，满足以下场景：

### 广播消息

* 系统公告
* VIP 公告
* 营销活动通知

特点：

* 一条消息面向大量用户
* 用户数量可达 1 亿+
* 采用读扩散（Fan-Out Read），不生成 user_message

---

### 用户消息

* 订单通知
* 物流通知
* 售后通知
* 风控通知

特点：

* 一条消息对应一个用户
* 强关联业务实体
* 采用写扩散（Fan-Out Write），直接写入 user_message

---

### 非功能性要求

* 支持 1 亿用户
* 支持水平扩展
* 支持 Kafka 异步投递
* 支持消息撤回（逻辑删除）
* 支持多端同步（用户维度 Cursor，不区分设备）
* 支持未读数统计
* 支持高并发查询
* Redis 不可用时降级到 MySQL

---

# 2. 核心设计原则

## 2.1 混合读写扩散

### 广播消息 — 读扩散（Fan-Out Read）

消息只存一份（message 表），用户查询时动态计算可见消息。

禁止：

```text
1条广播消息生成1亿条user_message记录
```

### 用户消息 — 写扩散（Fan-Out Write）

直接写入 user_message 表（status=UNREAD），用户在消息中心直接查询。

---

## 2.2 Feed 模型

按业务类型拆分消息流，每个 Feed 独立维护。系统中不存在全局消息流。

```text
SYSTEM     - 系统消息
VIP        - VIP 消息
MARKETING  - 营销消息
ORDER      - 订单消息
LOGISTICS  - 物流消息
RISK       - 风控消息
```

---

## 2.3 Cursor 定义为"全部已读"水位

Cursor 仅用于"全部标已读"操作，表示用户在该 Feed 中已读到的消息 ID 位置。

Cursor 单调递增，永不回退。

更新规则：

```text
newCursor = max(oldCursor, newCursor)
```

**Cursor 不参与未读数计算**。未读数基于逐条已读记录计算（见 5.3 节）。

---

## 2.4 逐条已读与全部已读分离

| 操作 | 实现方式 | 影响范围 |
|------|----------|----------|
| 逐条已读 | 写入 user_message (status=READ)，UNIQUE 幂等 | 仅影响该消息 |
| 全部已读 | 推进 user_feed_cursor.cursor 到 feed 最大 ID | 影响该 Feed 所有未读消息 |

---

# 3. 领域模型

## Message

消息主体，描述消息内容。广播消息和用户消息都先写入此表。按时间归档，无需分片。

状态：NORMAL → RECALL（不可逆），EXPIRE（自动过期）

---

## Feed

消息流，定义消息属于哪个可见范围。

6 种类型：SYSTEM / VIP / MARKETING / ORDER / LOGISTICS / RISK

---

## MessageFeedMapping

消息与 Feed 的多对多关联。一条消息可以属于多个 Feed。

---

## UserMessage

用户级消息记录，双用途：

- **BROADCAST 已读记录**：逐条已读时写入 (send_type=0, status=READ)
- **USER 消息**：发送时写入 (send_type=1, status=UNREAD)

UNIQUE(user_id, message_id) 保证消费幂等。

---

## UserFeedCursor

用户在各 Feed 中的已读位置（全部已读水位）。单调递增。

---

## Subscription

用户订阅关系，控制用户可以访问哪些 Feed。

例如普通用户只能访问 SYSTEM，VIP 用户可访问 SYSTEM + VIP。

---

### 唯一索引


| send_type | status  | 含义                                 |

| 0         | READ    | 广播消息被用户逐条已读               |

# 5. Redis 设计

## 5.1 用户 Cursor（Hash）

### Key

```text
user:{userId}:cursor
```

### Field → Value

```text
system    → 1200
vip       → 300
marketing → 88
order     → 42
logistics → 15
risk      → 7
```

### 过期时间

7 天。MySQL 为持久化存储，Redis 过期后可回查回写。

---

## 5.2 Feed 最大位置（String）

### Key

```text
feed:{feedType}:max_cursor
```

### Value

```text
1250
```

广播消息发送时更新。用于 markAllRead 获取 feed 最新的消息 ID 作为 cursor。

---

## 5.3 未读数计算

**核心逻辑**：cursor 不参与未读数计算。未读数基于 user_message 表的已读记录计算。

```text
broadcastTotal = COUNT(message WHERE send_type=0 AND status=NORMAL AND feed={feedType})
broadcastRead  = COUNT(user_message WHERE send_type=0 AND status=READ AND user={userId} AND feed={feedType})
broadcastUnread = max(broadcastTotal - broadcastRead, 0)

userUnread = COUNT(user_message WHERE send_type=1 AND status=UNREAD AND user={userId} AND feed={feedType})

totalUnread = broadcastUnread + userUnread
```

总未读：

```text
totalUnread = Σ(feedUnread)
```

---

## 5.4 Feed 数量缓存（String + 30s TTL）

### Key

```text
feed:{feedType}:count
```

### Value

广播消息总数。30 秒 TTL，缓存 MySQL 查询结果。

---

## 5.5 消息去重（String + 7天 TTL）

### Key

```text
notify:dedup:{messageId}:{userId}
```

用于 Redis 层幂等校验（渠道发送流程），7 天自动过期。

---

## 5.6 Redis 降级策略

所有 Redis 操作均包裹 try-catch，Redis 不可用时自动降级到 MySQL：

- cursor 读取：Redis miss → MySQL 查询 → 回写 Redis
- unread 统计：Redis 不可用时直接查 MySQL
- feed cursor 更新：Redis 不可用时仅写 MySQL
- 去重校验：Redis 不可用时跳过（依赖 MySQL UNIQUE KEY 兜底）

---

# 6. REST API 设计

## 6.1 消息中心 API

### GET /api/v1/notifications/unread

获取用户各 Feed 的未读数。

**参数**：`userId` (Long)

**响应**：
```json
{
  "data": [
    { "feedType": "system",    "feedName": "系统消息", "unreadCount": 5 },
    { "feedType": "vip",       "feedName": "VIP消息",  "unreadCount": 2 },
    { "feedType": "marketing", "feedName": "营销消息", "unreadCount": 0 }
  ]
}
```

---

### GET /api/v1/notifications/feed/{feedType}

查询某个 Feed 中的未读消息列表。

**参数**：
- `userId` (Long) — 用户 ID
- `feedType` (String) — Feed 类型
- `after` (Long, optional) — 游标分页，上一页最后一条 messageId。不传则从 user_cursor 开始
- `size` (int, default=20) — 每页条数

**查询逻辑**：
```sql
SELECT m.* FROM message m
JOIN message_feed_mapping mfm ON m.id = mfm.message_id
LEFT JOIN user_message um ON m.id = um.message_id AND um.user_id = ? AND um.status = 1
WHERE mfm.feed_type = ?
  AND m.id > ?  -- cursor
  AND m.status = 0  -- NORMAL
  AND (m.expire_time IS NULL OR m.expire_time > NOW())
  AND um.message_id IS NULL  -- 排除已逐条已读
ORDER BY m.id ASC
LIMIT ?
```

---

### POST /api/v1/notifications/read

逐条标记消息为已读。

**参数**：`userId` (Long), `messageId` (Long), `feedType` (String)

**实现**：INSERT INTO user_message ... ON DUPLICATE KEY UPDATE status=READ。不移动 cursor。

---

### POST /api/v1/notifications/cursor

将某个 Feed 全部标为已读。

**参数**：`userId` (Long), `feedType` (String)

**实现**：获取 feed 当前最大消息 ID 作为新 cursor，调用 updateCursor（GREATEST 保证单调递增）。

---

### GET /api/v1/notifications（旧接口，兼容保留）

分页查询用户消息列表。合并广播消息和用户消息，按时间降序。

**参数**：`userId` (Long), `page` (int, default=0), `size` (int, default=20)

---

## 6.2 上游消息 API

### POST /api/v1/upstream/message

上游系统提交单条消息。

**Body**：
```json
{
  "messageId": "uuid",
  "appId": "order-service",
  "templateCode": "order_paid",
  "sendType": "BROADCAST",
  "feedTypes": ["order"],
  "recipients": { "userIds": [1001, 1002] },
  "channels": ["in_app", "sms"],
  "params": { "orderId": "123456" },
  "category": "order",
  "priority": "high"
}
```

- **sendType=BROADCAST**：读扩散，仅写 message + mapping，不生成 user_message
- **sendType=USER**（默认）：写扩散，写入 user_message (status=UNREAD)，同时兼容渠道发送

---

### POST /api/v1/upstream/messages/batch

批量提交消息。逐条调用 processMessage。

---

## 6.3 消息撤回 API

### POST /api/v1/recall

逻辑删除消息。

**Body**：`{ "messageId": 100, "reason": "发送错误" }`

**实现**：UPDATE message SET status = RECALL WHERE id = ?

# 7. 消息发送流程

## 7.1 入口

上游消息支持两种入口：

1. **Kafka 消费**：topic `notification.upstream.raw`，由 UpstreamMessageConsumer 消费
2. **REST API**：`POST /api/v1/upstream/message`，由 UpstreamController 接收

两种入口最终都调用 `UpstreamMessageService.processMessage()`。

---

## 7.2 广播消息发送（读扩散）

```text
上游系统 → Kafka / REST API
  → UpstreamMessageService.processMessage
    → 校验模板 (TemplateService)
    → 渲染标题
    → 写入 message 表
    → 写入 message_feed_mapping
    → 更新 Redis feed:max_cursor
    → 完成（不生成 user_message）
```

用户查询时通过 LEFT JOIN user_message 过滤已逐条已读的消息。

---

## 7.3 用户消息发送（写扩散）

```text
上游系统 → Kafka / REST API
  → UpstreamMessageService.processMessage
    → 校验模板
    → 渲染标题 + 内容
    → 写入 message 表
    → 写入 message_feed_mapping
    → 遍历收件人列表：
      → Redis 去重校验 (setIfAbsent, 7天 TTL)
      → 写入 user_message (status=UNREAD, send_type=USER)
      → 写入 notification_message（兼容旧渠道）
      → 调用 channelSendService.sendAsync（异步渠道发送）
```

---

# 8. 消息查询流程

## 8.1 Feed 未读数查询

```text
GET /api/v1/notifications/unread?userId=1001
  → SubscriptionService 获取用户可访问的 Feed 列表
  → CursorService.getUnreadCounts（遍历每个 Feed）
    → messageService.countByFeedType（广播消息总数，走 Redis 30s 缓存）
    → countReadByUserAndFeed（已逐条已读的广播消息数）
    → broadcastUnread = total - read
    → countUnreadUserByFeed（USER 类型未读数）
    → feedUnread = broadcastUnread + userUnread
  → 返回各 Feed 未读数
```

## 8.2 Feed 消息列表查询

```text
GET /api/v1/notifications/feed/{feedType}?userId=1001&after=500
  → 获取 cursor（未传 after 时取 user_cursor，否则用 after）
  → MessageMapper.findUnreadByFeedAndCursor
    → JOIN message_feed_mapping 过滤 Feed
    → LEFT JOIN user_message 排除已逐条已读
    → WHERE m.id > cursor AND m.status = NORMAL AND 未过期
    → ORDER BY m.id ASC LIMIT size
  → 返回消息列表
```

# 10. Cursor 更新流程

## 全部已读

```text
客户端请求 POST /api/v1/notifications/cursor
  → NotificationQueryService.markAllRead
    → messageService.getFeedMaxCursor（Redis → MySQL 回查）
    → CursorService.updateCursor
      → MySQL: UPDATE ... SET cursor = GREATEST(cursor, newCursor)
      → Redis: Hash 写入，7 天过期
```

原则：Cursor 永远不能回退（GREATEST 保证）。

# 11. 消息撤回设计

采用逻辑删除，禁止 DELETE FROM message。

```sql
UPDATE message SET status = 'RECALL' WHERE id = ?
```

已撤回消息在所有查询中自动过滤（WHERE m.status = 0）。

客户端同步通过 Kafka topic `notification.message.recall` 通知。

---

# 12. 幂等方案

## 12.1 数据库层

user_message 表 UNIQUE KEY (user_id, message_id) 保证同一消息对同一用户只插入一次。

insertRead 使用 ON DUPLICATE KEY UPDATE 保证幂等：

```sql
INSERT INTO user_message (...) VALUES (...)
ON DUPLICATE KEY UPDATE status = 1, read_time = NOW()
```

## 12.2 Redis 层

渠道发送流程使用 Redis SETNX（setIfAbsent）做去重，7 天自动过期。

Redis 不可用时跳过去重，依赖 MySQL UNIQUE KEY 兜底。

---

# 13. 多端一致性

采用用户维度 Cursor，而非设备维度 Cursor。

```text
userId=1001
  system    → cursor=1200
  vip       → cursor=300
  marketing → cursor=88
```

所有设备（iPhone、Android、Web、PC）共享同一组 Cursor。

---

# 14. Redis 降级与高可用

所有 RedisTemplate 操作包裹 try-catch：

| 场景 | 降级行为 |
|------|----------|
| cursor 读取 | Redis miss → MySQL 查询 → 异步回写 Redis |
| cursor 更新 | Redis 失败 → 仅写 MySQL |
| feed count 查询 | Redis miss → MySQL 查询 → 回写 Redis（30s TTL） |
| feed max_cursor | Redis 不可用 → 直接查 MySQL MAX(id) |
| 消息去重 | Redis 不可用 → 跳过（MySQL UNIQUE KEY 兜底） |

# 17. 核心设计总结

1. 广播消息采用读扩散（Fan-Out Read），不生成 user_message
2. 用户消息采用写扩散（Fan-Out Write），写入 user_message (status=UNREAD)
3. 引入 Feed 模型管理消息流，6 种 Feed 独立维护
4. Cursor 仅用于"全部已读"水位，单调递增不回退
5. **Cursor 不参与未读数计算**，未读数基于逐条已读记录计算
6. Redis Hash 存储 Cursor（7 天 TTL），MySQL 为持久化存储
7. Feed 广播消息总数缓存 30 秒 TTL
8. 逐条已读写入 user_message（ON DUPLICATE KEY UPDATE 幂等）
9. 全部已读推进 cursor（GREATEST 保证单调递增）
10. 查询时 LEFT JOIN 排除已逐条已读的广播消息
11. Kafka 消费采用业务幂等（UNIQUE KEY + ON DUPLICATE KEY UPDATE）
12. 消息撤回采用逻辑删除（状态置为 RECALL）
13. 所有 Redis 操作均带 try-catch 降级，Redis 不可用时回退 MySQL
14. 多端共享用户维度 Cursor
15. RabbitMQ/WebSocket 实时同步（规划中，未实现）
16. 冷热数据分层存储（规划中，未实现）
